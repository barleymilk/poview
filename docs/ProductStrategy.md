# POVIEW 제품 전략

## 배경
- [ ] POVIEW(포뷰)는 "온라인에서 가구를 실제처럼 체감하기 어렵다"는 구조적 문제를 해결하는 FE 중심 제품 실험이다.
- [ ] 디지털 트윈 기반 쇼핑 경험은 차별화 가능성이 높지만, 초기 리소스 제약(에셋/시간/개발 난이도)을 고려한 집중 전략이 필요하다.
- [ ] 취업 포트폴리오 목적상, 범용 기능 확장보다 명확한 사용자 가치와 기술 완성도를 증명하는 전략이 우선이다.

## 목표
- [ ] 핵심 제품 목표를 `키 기반 1인칭 POV`로 고정하고, 사용자별 체감 차이를 시각적으로 전달한다.
- [ ] 비즈니스 목표는 `사이즈 미스 방지`와 `예산 기반 추천`으로 설정해 구매 의사결정의 실질적 개선을 목표로 한다.
- [ ] MVP에서 좁은 카테고리 집중으로 높은 데모 완성도와 일관된 UX를 확보한다.

## 범위
- [ ] 서비스명은 모든 문서/화면에서 `POVIEW`를 공식 표기한다.
- [ ] MVP 타겟 카테고리는 `홈 오피스(Desk-terior)` 단일 트랙으로 제한한다.
- [ ] MVP 포함 기능
- [ ] 1) 사용자 프로필 입력: 키, 예산, 공간 기본 치수
- [ ] 2) POV 시뮬레이션: 키 기반 카메라 높이/시야 기본값 적용
- [ ] 3) 공간 적합성 피드백: 배치 가능 여부, 하부 여유 공간 등 핵심 수치 표시
- [ ] 4) 예산 기반 추천: 예산 구간별 세트 구성(책상+의자+보조 아이템)
- [ ] MVP 제외 기능
- [ ] 1) 전 카테고리 확장
- [ ] 2) 실시간 물리 시뮬레이션
- [ ] 3) 과도한 소셜/커뮤니티 기능

## 작업 단계
- [ ] 1단계: 문제-가치 시나리오 정의
- [ ] 대표 사용자 시나리오를 수치 중심으로 작성한다. (예: 160cm 사용자, 책상 하부 여유 공간 확인)
- [ ] 2단계: 카탈로그 축소 및 에셋 확보
- [ ] 무료 GLB 우선 전략으로 홈 오피스 핵심 SKU 풀을 구성하고, 스케일 정규화 규칙을 정의한다.
- [ ] 3단계: POV 엔진 설계
- [ ] 키 입력값을 카메라 높이 및 UI 피드백과 연결하는 변환 규칙을 설계한다.
- [ ] 4단계: 추천 로직 설계
- [ ] 예산 구간별 세트 추천 규칙을 작성하고, 필수/선택 아이템 구조를 정의한다.
- [ ] 5단계: MVP 통합
- [ ] 제품 탐색 -> 시뮬레이션 -> 추천 비교 -> 의사결정 흐름을 하나의 UX로 통합한다.
- [ ] 6단계: 검증/개선
- [ ] 사용자 테스트 결과를 기반으로 POV 정확도 인식, 추천 신뢰도, 성능을 우선 개선한다.

## 리스크
- [ ] 에셋 리스크: 무료 GLB 라이선스/품질/치수 신뢰도 이슈
- [ ] 기술 리스크: 키 기반 POV가 기대만큼 직관적으로 해석되지 않을 가능성
- [ ] 제품 리스크: 사용자에게 "기능은 인상적이나 구매 전환과 연결이 약함"으로 평가될 가능성
- [ ] 일정 리스크: 3D/UX/추천 기능을 동시에 구현하면서 일정이 과도하게 늘어질 가능성

## 검증 계획
- [ ] POV 가치 검증
- [ ] "키가 다르면 체감이 달라진다"는 인식 여부를 시나리오 테스트로 확인한다.
- [ ] 여유 공간 수치 표기가 의사결정 신뢰도를 높이는지 인터뷰와 설문으로 검증한다.
- [ ] 추천 가치 검증
- [ ] 예산 기반 세트 추천이 단품 탐색 대비 의사결정 시간을 단축하는지 측정한다.
- [ ] 추천 수용률(추천 세트 클릭/비교/저장 지표)을 추적한다.
- [ ] 포트폴리오 검증
- [ ] 최종 데모에서 FE 기술 설명 가능 항목(렌더링 최적화, 상태 동기화, UX 설계 근거)을 체크리스트로 관리한다.
