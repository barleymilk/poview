# POVIEW 아이디어 평가

## 배경
- [ ] POVIEW(포뷰)는 가구/인테리어 제품을 구매하기 전에 사용자의 실제 시점에서 공간감을 검증하려는 문제의식에서 출발한다.
- [ ] 기존 이커머스는 제품 단독 이미지 중심이라 사용자 체형, 시점, 공간 제약(높이/폭/깊이)에 따른 사용성 판단이 어렵다.
- [ ] 취업용 FE 포트폴리오 관점에서, 시각화 기술(Web 3D)과 사용자 문제 해결(구매 의사결정 지원)을 동시에 증명할 수 있는 주제가 필요하다.

## 목표
- [ ] POVIEW의 핵심 가치(사이즈 미스 방지, 예산 기반 추천)가 실제 구매 의사결정에 유의미한 도움을 주는지 검증한다.
- [ ] 디지털 트윈 개념을 도입하되, 기술적 핵심 도전 과제를 `사용자 키(Height) 데이터 기반 1인칭 시점(POV) 구현`으로 명확히 정의한다.
- [ ] 단순 "보여주기"를 넘어 수치 기반 시나리오(예: 160cm 사용자 기준 다리 공간 여유)로 가치 전달 수준을 높인다.

## 범위
- [ ] 초기 도메인은 전 가구 카테고리가 아닌 `홈 오피스(Desk-terior)`로 제한한다.
- [ ] 현실적인 3D 에셋 전략으로 고퀄리티 무료 GLB 모델을 우선 활용하고, 부족한 항목은 대체 가능한 유사 모델로 매핑한다.
- [ ] 핵심 기능 범위는 아래로 제한한다.
- [ ] 1) 사용자 키 입력 및 POV 카메라 높이 반영
- [ ] 2) 제품 배치 시 충돌/간섭 및 여유 공간 시각화
- [ ] 3) 예산 구간 기반 추천 세트(저/중/고 예산)
- [ ] 4) 카테고리 내 대표 시나리오 검증(책상+의자+수납 중심)

## 작업 단계
- [ ] 1단계: 문제 정의 고도화
- [ ] 사용자 페르소나(키/예산/공간 제약)와 주요 실패 시나리오(사이즈 미스, 동선 불편)를 문서화한다.
- [ ] 2단계: 가치 가설 수립
- [ ] "POV 시뮬레이션이 반품 가능성 판단 시간을 줄인다" 같은 측정 가능한 가설을 정의한다.
- [ ] 3단계: 기술 타당성 검증
- [ ] Web 기반 3D 렌더링에서 키 기반 카메라 모델, 성능, 로딩 전략의 최소 구현 가능성을 확인한다.
- [ ] 4단계: 범위 고정
- [ ] 홈 오피스 내 우선 상품군(책상/의자/모니터암/서랍장)을 확정하고 필수 GLB 리스트를 만든다.
- [ ] 5단계: 검증 시나리오 설계
- [ ] "160cm 사용자 기준 책상 하부 여유 공간 시각화" 등 수치 기반 데모 시나리오를 작성한다.

## 리스크
- [ ] 3D 에셋 품질 편차: 무료 GLB의 스케일/축/재질 품질이 일관되지 않을 수 있다.
- [ ] 성능 리스크: 저사양 기기에서 렌더링 품질과 프레임 저하가 발생할 수 있다.
- [ ] 정확도 리스크: 실제 치수와 모델 치수 불일치 시 신뢰도 하락이 발생한다.
- [ ] 사용자 기대치 리스크: "정확한 물리 시뮬레이션"으로 오해되면 경험-현실 간 차이로 실망 가능성이 있다.

## 검증 계획
- [ ] 원칙: 실제 대규모 유저 없이도 `성능 + 사용성 시뮬레이션 + 소규모 인터뷰`로 검증한다.
- [ ] 1) 정량 검증 - 성능(Web Vitals + 3D 최적화)
- [ ] 목표: 3D 라이브러리와 고화질 모델 사용 환경에서도 쾌적한 체험을 유지한다.
- [ ] 방법: Lighthouse로 Web Vitals를 측정하고, `stats.js`로 실시간 FPS를 기록한다.
- [ ] 측정 지표: LCP, 평균 FPS(모바일 기준), GLB 모델 용량(Draco 압축 전/후), 첫 유효 화면까지 시간.
- [ ] 합격 기준: LCP 2.5초 이내, 모바일 평균 60FPS 근접 유지(최저 45FPS 이상), Draco 압축 후 용량 60% 이상 감소 목표(권장 80%), 치명적 프레임 드랍 0건.
- [ ] 2) 정량 검증 - 결정 효율성 Self A/B 테스트
- [ ] 목표: POV 시뮬레이션이 이미지/스펙 중심 화면보다 의사결정 시간을 줄이는지 확인한다.
- [ ] 방법: 지인 5명 대상 `A(이미지+스펙)`와 `B(POVIEW POV 모드)`를 같은 과제로 비교 테스트한다.
- [ ] 테스트 질문 예시: "180cm 사용자가 이 책상을 사용할 때 낮게 느껴질 가능성이 있는가?"
- [ ] 측정 지표: 질문당 응답 시간(초), 오답률(사이즈 오판), 확신도 점수(5점 척도).
- [ ] 합격 기준: B안의 평균 응답 시간 20% 이상 단축, 오답률 30% 이상 감소, 확신도 평균 3.5/5 이상.
- [ ] 3) 정성 검증 - 신뢰 형성 요인 인터뷰
- [ ] 목표: 사용자가 어떤 시각 요소에서 "실물에 가깝다"고 느끼는지 식별한다.
- [ ] 방법: 지인 3명 대상 15분 심층 인터뷰를 진행하고 발화 내용을 기록한다.
- [ ] 관찰 포인트: 바닥 그리드(Grid), 그림자(Shadow), 실시간 수치 표기 중 신뢰도에 가장 크게 기여한 요소.
- [ ] 합격 기준: 신뢰 형성 핵심 요소 상위 2개 도출, 각 요소의 개선 액션 1개 이상 정의.
- [ ] 4) 정성 검증 - Edge Case 및 한계 인식
- [ ] 목표: POV 시뮬레이션이 제공하지 못하는 정보의 경계를 명확히 문서화한다.
- [ ] 방법: 시뮬레이션 경험과 실제 물리 경험의 차이를 사례 중심으로 기록한다.
- [ ] 예시: 시야/치수 판단은 가능하지만 쿠션감, 재질 촉감 등 촉각 정보는 대체 불가.
- [ ] 합격 기준: 한계점 3개 이상과 보완 방향(텍스트 고지, 리뷰 연동, 소재 설명 강화) 정의.
- [ ] 5) 포트폴리오 검증(스프린트 종료 시)
- [ ] 결과를 `문제-가설-실험-결론` 구조로 정리하고, 수치 스크린샷/영상 링크를 함께 남긴다.
- [ ] 합격 기준: FE 기술 의사결정 근거 2개 이상(성능 최적화 1개 + UX 개선 1개) 설명 가능 상태 확보.
